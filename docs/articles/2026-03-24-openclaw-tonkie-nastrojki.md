# OpenClaw для Telegram-канала: тонкая настройка рабочего контура без хрупких мест

Опубликовано: https://telegra.ph/OpenClaw-dlya-Telegram-kanala-tonkaya-nastrojka-rabochego-kontura-03-24

Эта статья — подробный практический разбор того, как собрать в OpenClaw устойчивый контур управления Telegram-каналом: от транспорта и форматирования до редакционной политики и long-form публикаций через Telegra.ph.

Задача не в том, чтобы «автоматизировать всё», а в том, чтобы автоматизировать только повторяемое и рискованное, сохранив контроль в критических точках.

## 1. От чего мы отталкиваемся: базовая архитектура процесса

Рабочая модель состоит из четырёх слоёв:

1. **Транспортный слой** — надёжная доставка сообщений в канал через Maton gateway.
2. **Форматирующий слой** — стабильный HTML-safe вывод для Telegram без markdown-артефактов.
3. **Редакционный слой** — правила подготовки и согласования контента.
4. **Long-form слой** — подробные статьи в Telegra.ph, связанные с короткими постами.

Почему это важно: если пропустить хотя бы один слой, система выглядит «почти рабочей», но начинает сбоить в реальной эксплуатации.

## 2. Транспортный слой: что должно быть настроено в первую очередь

### 2.1 Проверенные параметры

В текущей рабочей конфигурации критичны три переменные:

- `MATON_API_KEY`
- `TELEGRAM_BOT_TOKEN`
- права бота в канале (`@algorithmic_planetoid_bot` как администратор)

Проверенный маршрут отправки:

- `https://gateway.maton.ai/telegram/bot<TOKEN>/<method>`
- токен передаётся в URL-encoded виде.

Практический вывод: шаблон `:token` может быть нерабочим в конкретной среде, поэтому маршрут нужно валидировать фактически, а не по «ожидаемому» паттерну.

### 2.2 Минимальная диагностика перед боевым использованием

До публикаций обязательно проходят:

- `getMe` (аутентификация бота)
- `getChat` (доступ к каналу)
- dry-run публикации (валидация payload)

Если любой из шагов падает, публикационный поток не должен продолжаться.

## 3. Форматирование Telegram: красиво, но технологично

### 3.1 Почему не Markdown

В Bot API Markdown/MarkdownV2 формально поддерживаются, но в реальном pipeline часто вызывают артефакты из-за экранирования. Для стабильного канального процесса выбран другой приоритет:

- Markdown/MarkdownV2 отключены в рабочем контуре;
- по умолчанию используется `parse_mode=HTML`;
- plain text остаётся fallback-режимом.

### 3.2 Как ограничить HTML до безопасного подмножества

Разрешён только управляемый набор тегов (`b/strong`, `i/em`, `u`, `s`, `a`, `code`, `pre`, `blockquote`, `tg-spoiler`) и обязательное экранирование спецсимволов вне тегов.

Это позволяет сделать текст визуально выразительным, но предсказуемым для рендера Telegram.

### 3.3 Emoji-политика

Пост не должен выглядеть как «гирлянда». Рабочий стандарт:

- обычно 1–3 emoji на пост;
- emoji используются только как смысловые акценты (hook, контраст, CTA);
- техническая аргументация всегда остаётся первичной.

## 4. Редакционный протокол: где нужен человек

Безопасная автоматизация не означает «без человека». Она означает «человек в нужной точке».

Обязательный процесс:

- `draft -> approval -> publish`
- destructive-операции запрещены по умолчанию
- если запрос: «пост со статьёй» — сначала утверждается статья, потом готовится пост

Эта последовательность защищает от ситуаций, когда короткий пост уже опубликован, а long-form материал ещё не стабилен по смыслу.

## 5. Long-form через Telegra.ph: как избежать артефактов

### 5.1 Ключевой момент формата

Telegra.ph API принимает не markdown parse mode, а `content` в формате Node JSON. Поэтому надёжный путь — детерминированная конверсия текста в узлы, а не «прямая отправка markdown». 

### 5.2 Что уже автоматизировано в Python

В рабочем контуре уже есть скрипты:

- `telegraph_create_account.py`
- `telegraph_publish.py` (dry-run по умолчанию)
- `telegraph_edit.py`
- `telegraph_views.py`
- `telegraph_common.py` (конверсия markdown -> Node JSON)

Дополнительно реализованы:

- конверсия inline-форматирования (`bold/italic/code/strike/links`)
- ordered/unordered lists
- поддержка heading-модели с маппингом к поддерживаемым тегам Telegraph

## 6. Skills-стек: как убрать эффект «нечеловеческого текста»

Для канала и статей используется комбинированный стек:

- базовые: `telegram-channel-copywriter`, `telegraph-publisher`
- усилители: `copywriter`, `social-media-scheduler`
- анти-uncanny слой: `humanize`, `human-writing`, `writer`

В практическом режиме это даёт два эффекта:

1. стабильную структуру и техническую ясность;
2. более естественный ритм повествования без роботических повторов.

## 7. Публичный GitHub-репозиторий: где лежит воспроизводимая версия

Чтобы все ссылки и практики из статьи были доступны внешне, собран отдельный публичный проект с копиями рабочих артефактов:

- https://github.com/remote-worker-prof/openclaw-channel-automation-public

В репозитории вынесены:

- статьи и плейбуки;
- публично безопасные скрипты;
- кастомные skills для копирайтинга и Telegra.ph;
- архив публикационных пар (статья + пост + метаданные).

Перед публикацией репозитория выполняется отдельный секрет-скан, чтобы исключить утечки токенов, ключей и приватных переменных окружения.

## 8. Организация контента в KnowledgeVault

Чтобы публикации не терялись, хранение должно быть операционно удобным. Для каналов выделяется отдельный раздел, где каждая публикация ведётся в собственной папке:

- `<ISO-date>-<post-title-slug>/`

Внутри хранятся как минимум:

- `article.md`
- `article-url.txt`
- `post.html` (или `post.txt`)
- `post-meta.md` (id сообщения, дата публикации, заметки)

Эта схема важна не только для порядка, но и для повторного использования контента, ретроспектив и аналитики.

## 9. Тонкие настройки, которые чаще всего забывают

1. **Не смешивать форматные режимы** в одной цепочке (Markdown и HTML вместе).
2. **Не пропускать dry-run** перед любой внешней публикацией.
3. **Не хранить рабочие черновики только во временной staging-зоне**.
4. **Не публиковать пост до утверждения статьи**, если контент связан.
5. **Не “улучшать стиль” за счёт потери смысла**: приоритет всегда у точности.

## 10. Проверочный чеклист перед выпуском

Перед отправкой в канал проверяются:

- валидность ссылки на статью;
- непротиворечивость «пост ↔ статья»;
- корректный формат (HTML-safe, без markdown-маркеров);
- умеренность emoji;
- финальное подтверждение публикации.

## Заключение

Зрелая автоматизация канала — это дисциплина, а не набор API-вызовов. Когда транспорт, форматирование, редакционный протокол и long-form слой собраны в единую систему, канал получает одновременно скорость, качество и управляемость.

Именно такая комбинация даёт долгосрочный эффект: публикации становятся не только регулярными, но и предсказуемо полезными для читателя.
